## 1. Design System Foundation

### Aesthetic Direction: "Refined Developer Studio"

Not cyberpunk. Not brutalist. Not generic SaaS. This is a **precision instrument for developers** â€” think VS Code meets Notion meets iA Writer. The aesthetic is:

- **Intentional minimalism** â€” every pixel earns its place
- **Warm dark mode as default** â€” not pure black (#000), warm charcoal (#1a1b26)
- **Surgical color accents** â€” one dominant accent per theme, never competing
- **Typographic hierarchy as the primary design element** â€” the text IS the product
- **Depth through subtle layering** â€” not flat, not skeuomorphic, but layered with purpose

## 3. Typography Overhaul

### Current Problem

The app uses generic system fonts (`system-ui, -apple-system, sans-serif`) for UI and (`Consolas, Monaco, Courier New`) for the editor. These are safe but forgettable â€” they make the app look like a browser default.

### Recommended Font Pairing: **"Developer Mono"**

| Role | Font | Weights | Usage |
|------|------|---------|-------|
| **UI / Body** | IBM Plex Sans | 300, 400, 500, 600, 700 | Toolbar labels, sidebar text, modal content, footer |
| **Headings** | JetBrains Mono | 500, 600, 700 | App title, section headers, feature labels |
| **Editor** | JetBrains Mono | 400, 500 | Markdown editor textarea |
| **Preview Code** | JetBrains Mono | 400 | Code blocks in rendered markdown |
| **Preview Body** | IBM Plex Sans | 400, 500 | Rendered markdown prose |

**Why this pairing:**
- JetBrains Mono is the gold standard for code editors â€” ligatures, clear distinction between similar characters (0/O, 1/l/I)
- IBM Plex Sans is the most readable humanist sans-serif designed specifically for developer UIs (used by IBM Carbon Design System)
- Both have excellent Unicode coverage, mathematical symbols, and icon support
- Both are free, open-source, and on Google Fonts

### CSS Implementation

```css
/* Load via Google Fonts (add to index.html <head>) */
@import url('https://fonts.googleapis.com/css2?family=IBM+Plex+Sans:wght@300;400;500;600;700&family=JetBrains+Mono:wght@400;500;600;700&display=swap');

/* Typography scale (based on 1.25 ratio) */
:root {
  --font-ui: 'IBM Plex Sans', -apple-system, BlinkMacSystemFont, sans-serif;
  --font-mono: 'JetBrains Mono', 'Fira Code', 'Consolas', monospace;

  --text-xs: 0.75rem;    /* 12px â€” captions, badges */
  --text-sm: 0.875rem;   /* 14px â€” secondary text, sidebar */
  --text-base: 1rem;     /* 16px â€” body text */
  --text-lg: 1.125rem;   /* 18px â€” emphasized body */
  --text-xl: 1.25rem;    /* 20px â€” section titles */
  --text-2xl: 1.5rem;    /* 24px â€” page titles */
  --text-3xl: 1.875rem;  /* 30px â€” hero text */

  --leading-tight: 1.25;   /* Headings */
  --leading-normal: 1.5;   /* UI text */
  --leading-relaxed: 1.75; /* Long-form reading (preview) */

/* Application */
```

## 4. Color System Redesign

### Current State

Your theme system is already excellent â€” 25+ CSS variables per theme, 7 theme families. The problem isn't the themes themselves but the **base semantic layer** that the themes plug into.

### Proposed Semantic Color Architecture

Every theme must provide these semantic tokens. The app references ONLY semantic tokens, never raw hex:

```css
:root {
  /* === SURFACE LAYERS (background hierarchy) === */
  --surface-0: /* deepest background (app shell) */;
  --surface-1: /* primary panels (editor, preview) */;
  --surface-2: /* elevated elements (cards, dropdowns) */;
  --surface-3: /* highest elevation (tooltips, popovers) */;

  /* === TEXT HIERARCHY === */
  --text-primary: /* main content â€” contrast â‰¥ 7:1 on surface-1 */;
  --text-secondary: /* descriptions, labels â€” contrast â‰¥ 4.5:1 */;
  --text-tertiary: /* placeholders, disabled â€” contrast â‰¥ 3:1 */;
  --text-inverse: /* text on accent/CTA backgrounds */;

  /* === INTERACTIVE STATES === */
  --accent: /* primary brand/action color */;
  --accent-hover: /* hover state of accent */;
  --accent-active: /* active/pressed state */;
  --accent-subtle: /* low-emphasis accent backgrounds */;
  --accent-rgb: /* RGB triplet for rgba() usage, e.g., 59, 130, 246 */;

  /* === SEMANTIC FEEDBACK === */
  --success: #22c55e;
  --success-subtle: rgba(34, 197, 94, 0.1);
  --warning: #f59e0b;
  --warning-subtle: rgba(245, 158, 11, 0.1);

  --error-subtle: rgba(239, 68, 68, 0.1);
  --info: #3b82f6;

  /* === BORDERS & DIVIDERS === */
  --border-default: /* subtle divider lines */;
  --border-strong: /* emphasized borders (focus, active) */;
  --border-interactive: /* borders on interactive elements */;
```

### Performance Rules

1. **ONLY animate `transform` and `opacity`** â€” never width, height, top, left, margin, padding
2. **Use `will-change` sparingly** â€” only on elements that actively animate
3. **Avoid `transition: all`** â€” explicitly list properties: `transition: background var(--duration-fast), color var(--duration-fast)`
4. **Cap animation duration at 400ms** for UI elements â€” longer feels sluggish
